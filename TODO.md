### 1. 강좌요금표 강사료/수용비 수동 조정 시 덮어써지던 문제 — 해결됨
- **현상이었던 것**: 강좌요금표에서 강사료·수용비를 수동으로 조정한 뒤, 차수별시수(결강 등)를 다시 조정하면 `updateC()`가 부서마스터 값으로 강사료·수용비를 덮어써서 수동 조정분(인상분 등)이 사라짐.
- **적용한 해결책**: "기준점(anchor) 저장" 방식을 [app-ui-steps.js:295-326](app-ui-steps.js:295) `updateC()`에 구현.
  1. 강사료(`instTot`)/수용비(`mgmtTot`)를 직접 수정하는 순간, `window.C[nm][q]._instAnchor = {amt, th}` / `_mgmtAnchor = {amt, th}`로 그 시점의 (금액, 총시수)를 저장.
  2. 이후 `mh`나 `unit`이 바뀌면, 앵커가 있는 쪽은 항상 **이 저장된 기준점에서부터** `amt × (새 tH / 앵커 tH)`로 비례 재계산 (부서마스터로 덮어쓰지 않음). 앵커가 없는 쪽(한 번도 수동 조정 안 한 값)은 기존처럼 부서마스터 기준으로 자동 계산.
  3. `resetC()`(마스터 기준으로 복구)는 `regenerateC()`에서 새 객체를 만들며 앵커 필드를 자연스럽게 버림 → 별도 처리 불필요.
  4. 브라우저에서 직접 검증: 가상마술(A) 강사료를 105,000→110,000으로 수동 인상 후 mh를 4,4,4→4,3,4로 변경 → instTot이 부활(96,250)하지 않고 100,830(비례 유지)으로 정확히 계산됨. mgmtTot도 별도 앵커로 독립 동작 확인. 초기화 버튼으로 앵커가 깨끗이 지워지는 것도 확인.
  5. 반올림 함수는 일단 기존과 동일하게 `Math.round` 유지 (강사료=버림/수용비=올림으로 바꾸는 건 별개 정책 결정이라 보류 — 아래 2번 참고).

### 2. [완료] 분기 강사료=내림(`Math.floor`), 수용비=올림(`Math.ceil`) 적용
- 사용자가 기억하기로 예전엔 강사료는 내림, 수익자부담(자부담)은 올림으로 계산했던 것 같다고 해서 시작된 논의. "수익자부담(청구서 안분, [app-ui-export.js])"이 아니라 **분기 강사료·수용비 자체(강좌요금표)**에 적용하기로 최종 정리함.
- **적용 지점**: [app-ui-steps.js](app-ui-steps.js) 두 곳 — `regenerateC()`(부서마스터 → 강좌요금표 자동계산, `qI`/`qM`)와 `updateC()`의 mh/unit 변경 시 재계산(`nextInstTot`/`nextMgmtTot`, 앵커 있는 경우·없는 경우 둘 다). 강사료 쪽은 전부 `Math.floor`, 수용비 쪽은 전부 `Math.ceil`로 교체.
- **근거**(이전 대화 결론): 강사료를 버림으로 하면 강사에게 계약금 비례분보다 더 나가는 일이 절대 없고(안전), 수용비를 올림으로 하면 운영비가 부족해지는 일이 없음. 둘을 합친 총액이 정확한 비례 총액보다 아주 소액(한 자릿수~10원) 많아질 수 있는 정도의 트레이드오프는 감수하기로 함.
- **브라우저에서 검증**: 가상마술(A)에 강사료 110,000/수용비 3,500 앵커를 걸어두고(th=12) mh 합계를 13으로 바꿨을 때, `Math.round` 기준이었다면 강사료 119,170/수용비 3,790이 나왔을 것을 `Math.floor`/`Math.ceil` 적용 후 **강사료 119,160(버림 확인)/수용비 3,800(올림 확인)**으로 정확히 달라짐을 확인. 앵커 없는 자동계산 경로(로봇과학(A))도 에러 없이 정상 동작 확인.
- **아직 미반영(범위 밖으로 남겨둠)**: [app-ui-export.js](app-ui-export.js)의 청구서 안분(`sT_i`/`tc_i`/`tf_i` 등, 강사료 몫을 `Math.round`로 계산 후 수용비 몫은 나머지로 유도하는 부분)은 이번 변경에 포함 안 됨 — 이건 "강좌요금표의 분기 강사료/수용비"가 아니라 "이미 확정된 학생별 청구액을 강사료/수용비로 쪼개는" 별개 계산이라 그대로 둠. 필요하면 여기도 강사료 몫 계산을 `Math.floor`로 바꾸는 걸 추가로 논의.

### 3. 수용비는 강사료의 5% 이내여야 한다는 행정 규정 — 하드 블록 시도 후 표시 전용으로 최종 정착
- **배경**: 반올림/버림 방식과 무관하게, "수용비 ≤ 강사료 × 5%" 라는 별도의 행정 규정이 있음. 튜토리얼 샘플 데이터 중 로봇과학(5.71%)·생명과학(5.26%)이 이미 이 규정을 위반하고 있었음.
- **1차로 하드 블록을 구현·검증**했으나(위반 시 `window.showAlert` + 입력값 되돌림, `commitState` 자체를 생성 안 함), 아래 4번(비율·한도 실시간 표시)까지 추가되고 나니 사용자가 "확인창이 너무 많아지고, 데이터 입력 시 이미 뜨는 하단 알림과 겹쳐 시선이 분산된다"고 판단 → **하드 블록은 제거하고, 4번의 실시간 표시로만 안내하는 것으로 최종 결정**.
- 제거 내역: [app-ui-steps.js](app-ui-steps.js) `updateM()`/`updateC()`에서 `checkMgmtRatio` 기반 차단 분기(alert+return)를 삭제. `window.MGMT_RATIO_LIMIT`/`window.checkMgmtRatio`는 4번의 실시간 표시가 계속 사용하므로 그대로 유지.
- 브라우저에서 재검증: 로봇과학 월수용비 2,000→3,000(8.6%, 초과) 입력이 **차단 없이 정상 저장**되고, 비율 표시만 빨간색 "⚠초과"로 갱신됨. 콘솔 에러 없음.

### 5. [완료] 부서 완전삭제 — 마감된 청구 이력이 있으면 삭제 자체를 하드 블록
- **배경**: "삭제"는 부서명 오타 등 실사용 전 데이터를 완전히 지우는 기능, "운영/미운영 체크"는 이미 운영 중이던 부서를 폐강 처리하는 기능 — 서로 의미가 다름(사용자 확인). `toggleDeptActive()`는 `isActive` 플래그만 바꾸고 `inst_m`/`mgmt_m`/`instTot`/`mgmtTot`는 안 건드리므로 애초에 안전함.
- **처음엔 "강한 경고문"으로 구현했었으나, 사용자가 실 데이터로 직접 재현해본 결과 경고 문구 자체가 틀렸다는 게 드러남**:
  - 실제로는 "배분만 부정확해지는" 정도가 아니라, **청구서에서 그 부서가 통째로 사라짐.**
  - 원인: `window.Hs`는 매 재계산(`autoRunSet()`, [app-engine.js:88](app-engine.js:88) `window.Hs = []`)마다 `window.E`로부터 처음부터 다시 만들어지는 파생 데이터. `delDept()`가 `e.course`를 `'미배정(누락)'`로 바꿔버리면, `window.SysSet.closedSess`에 저장된 마감 스냅샷의 조회 키(`` `${학생id}_${강좌명}` ``, [app-engine.js:156](app-engine.js:156))가 더 이상 매칭되지 않아 그 학생의 그 분기 금액이 통째로 0으로 재계산됨 → 청구서의 `if (d.sT === 0 && d.finT === 0) return;` 필터에 걸려 화면에서 사라짐.
  - 즉 "총액은 유지, 배분만 깨짐"이 아니라 **"마감된 자료를 포함해 부서 관련 데이터 자체가 사라짐"** 이 맞는 설명.
- **최종 결론(사용자 제안)**: 경고로는 부족하다 — 애초에 **마감된 이력이 있으면 삭제 자체를 막는 게 맞다.** `updateM()`/`updateC()`가 `isQuarterLocked()`로 편집을 막는 것과 같은 방향.
- **최종 구현**: [app-ui-steps.js](app-ui-steps.js)에 `deptHasClosedHistory(dept)` 헬퍼 추가 — `window.SysSet.closedSess`의 모든 스냅샷을 순회하며, 그 안의 학생별 키(`` `${id}_${강좌명}` ``)에서 강좌명 부분이 이 부서 소속인지 검사(마지막 `_` 기준으로 분리). `delDept()`는 이 체크가 true면 `window.showAlert`로 **하드 블록**(삭제 진행 없이 즉시 return) — "운영 체크 해제를 쓰거나, 정 삭제하려면 4스텝에서 관련 분기 마감을 먼저 해제하라"고 안내. 이력이 없는 부서는 기존과 동일하게 정상적인 확인창 흐름(취소 가능) 유지.
  - `window.Hs`가 아니라 `closedSess` 스냅샷을 직접 검사하는 이유: `window.Hs`는 마감 여부와 무관하게 현재 분기의 살아있는 등록생도 전부 포함하는 파생 데이터라, 이걸로 판단하면 "아직 마감 안 된 재학생만 있는" 부서까지 지나치게 광범위하게 막힘. 실제로 되돌릴 수 없는 건 `closedSess`에 박제된 마감 스냅샷뿐이므로, 정확히 거기만 검사함.
- **브라우저에서 검증**: 1분기 1차를 실제로 마감(`toggleSessCheck`)해 `closedSess['1_0']`에 진짜 스냅샷 생성 → 이 상태에서 "가상마술" 삭제 시도 → 🚫 하드 블록 알림 정확히 노출, `window.M['가상마술']` 삭제 안 됨 확인. 이력 없는 신규 테스트부서는 기존 확인창 → 확인 시 정상 삭제됨 확인. 콘솔 에러 없음. 테스트용 마감은 다시 해제해서 원상 복구.
- **문구 수정(사용자 피드백)**: 초안 문구가 어색하다는 피드백을 받아 최종적으로 "분기(차수)가 마감된 부서는 삭제할 수 없습니다. '삭제'는 시스템에서 해당 부서를 완전히 지우는 행위입니다. 부서 폐강을 원하신다면, 삭제 대신 해당 부서나 강좌의 '운영' 체크박스를 해제해서 사용해 주세요. 부서 폐강이나 삭제를 원하신다면, 먼저 [4스텝 정산 마감]에서 관련 분기의 마감을 해제하셔야 합니다."로 정리함 — "삭제=완전삭제, 운영체크=폐강"이라는 두 기능의 의미 차이를 먼저 명확히 하고, 마감 해제가 선행 조건임을 안내하는 구조.
